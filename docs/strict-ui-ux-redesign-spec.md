# 睡了么 UI/UX 严格重做规范

版本：2026-05-14
目标：让任何 AI 或开发者按本文件重做原生 APK UI 时，能够稳定产出符合参考图气质、适合中老年人、功能极简、按钮明确的界面。未满足验收标准的实现必须返工。

## 1. 总目标

睡了么不是设置面板，也不是医疗诊断软件。它是一个温暖、可爱、开箱即用的睡眠守护和陪伴工具。

最终 UI 必须像参考图一样：

- 温馨亲切：奶油底、亮蓝、健康绿、晨间橙、警醒红。
- 大字体：中老年人不用眯眼看。
- 大按钮：半夜或刚醒时也能点准。
- 少选择：每个页面只突出一个当前任务。
- 有陪伴感：狗熊和小助手要像“在身边”，不是冷冰冰的设置项。
- 不诊断：只说“疑似异常、建议、提醒、陪伴”，不得出现“诊断、治疗、确诊”等承诺。

## 2. 不可违反的设计原则

以下规则是硬性验收项，违反任意一条即不通过。

1. 每个页面第一屏只能有一个主操作按钮。
2. 睡前首页第一屏不得出现超过 4 个可点击项。
3. 首页不得展示长篇检测解释、数据库摘要、技术可信度说明。
4. “开始守护”必须是首页最醒目的按钮，宽度接近全屏，高度不小于 88dp。
5. “停止守护”必须用红色，不能和“开始守护”共用蓝色。
6. “联系人未设置”必须在首页第一屏可见，文案必须写明风险：`未设家人电话`。
7. 小助手聊天页必须打开即进入听取状态，不得要求用户先输入文字或点击发送。
8. 小助手聊天页不得出现“AI、机器人、大模型”字样，使用“小助手、我、我在听”。
9. 强唤醒页必须满屏高对比，主按钮必须是红色 `我醒了`，不得小于 92dp 高。
10. 晨间页必须在第一屏显示 `我喝水了` 和 `已吃药` 两个大卡片。
11. 设置项不得出现在首页第一屏；普通用户不应被设置打断。
12. 所有健康相关文字必须保留非诊断边界：`这是提醒，不是医疗诊断` 或等价短句。
13. 所有页面主文本不得小于 22sp；辅助说明不得小于 18sp；按钮文字不得小于 24sp。
14. 页面中不允许出现一整屏只有文字说明的状态，必须配图或大按钮。
15. 不允许使用参考图整张截图裁切为页面背景；必须用独立生成的 logo、场景、人物、按钮资产。

## 3. 设计系统

### 3.1 色彩

必须统一使用以下颜色。除非是设计模型生成的插画内部颜色，代码 UI 不得随意新增主色。

| 用途 | 名称 | Hex | 使用位置 |
| --- | --- | --- | --- |
| 页面底色 | Warm White | `#FFF9EB` | 所有主页面背景 |
| 卡片白 | Card White | `#FFFFFF` | 内容卡片 |
| 主按钮蓝 | Guard Blue | `#2B68E2` | 开始守护、首页守护 Tab |
| 深蓝文字 | Text Navy | `#1C2D49` | 标题、主正文 |
| 辅助灰蓝 | Muted Slate | `#6B768C` | 次说明 |
| 健康绿 | Care Green | `#47A858` | 已就绪、喝水、我没事 |
| 晨间橙 | Morning Orange | `#FF931E` | 早安、吃药、晨间提示 |
| 警醒红 | Wake Red | `#EB4634` | 强唤醒、停止守护、我醒了 |
| 柔粉 | Companion Pink | `#FF8A8A` | 贴心小妹 |
| 柔紫 | Gentle Purple | `#9D8BE8` | 温柔姐姐 |

按钮和卡片允许使用同色浅渐变，但必须遵守：

- 蓝色按钮：顶部 `#5B8CFF`，底部 `#2B68E2`，边框 `#1C4BA6`。
- 红色按钮：顶部 `#FF6A5C`，底部 `#EB4634`，边框 `#B52B22`。
- 绿色卡片：背景从 `#FFFFFF` 到 `#EAF7EC`，边框 `#A7DDB1`。
- 橙色卡片：背景从 `#FFFFFF` 到 `#FFF0DA`，边框 `#FFC37C`。
- 蓝色卡片：背景从 `#FFFFFF` 到 `#EAF1FF`，边框 `#A6C1FF`。

### 3.2 字体

Android 原生实现使用系统默认中文字体，不引入额外字体文件。必须使用粗体建立层级。

| 类型 | 字号 | 字重 | 行高 | 用途 |
| --- | --- | --- | --- | --- |
| 页面主标题 | 34sp | Bold | 1.08 | `睡了么`、`早安护理`、小助手名字 |
| 卡片标题 | 28sp | Bold | 1.08 | 首页状态、晨间卡标题 |
| 主按钮 | 28sp | Bold | 1.00 | 开始守护、我醒了 |
| 二级按钮 | 24sp | Bold | 1.00 | 我喝水了、已吃药、看一眼 |
| 正文 | 22sp | Normal/Bold | 1.14 | 小助手说话、提醒正文 |
| 辅助说明 | 18sp | Normal | 1.16 | 风险说明、边界说明 |
| Tab 文案 | 20sp | Bold | 1.00 | 守护、早安、我的 |

验收方式：用模拟器截图检查，不得出现按钮文字被截断、长句挤出卡片、正文小到难读。

### 3.3 圆角和阴影

参考图是软糖感、玩具感，不是企业后台风。

- 页面大卡片圆角：28dp。
- 小信息卡圆角：24dp。
- 主按钮圆角：28dp，高度越大圆角越大，但不得做成完全胶囊除非参考图同类按钮。
- 底部导航外框圆角：28dp。
- 阴影：只用于主按钮和大卡片，2-4dp elevation；不得出现厚重黑阴影。

### 3.4 间距

统一按 4dp 网格。

- 页面左右边距：20dp。
- 顶部安全区后第一元素上边距：20dp。
- 卡片内边距：18-24dp。
- 卡片之间：14-18dp。
- 主按钮上下间距：16dp。
- 底部导航高度：112-128dp。

## 5. 页面总结构

App 只保留三个底部主 Tab：

1. `守护`
2. `早安`
3. `我的`

`小助手` 不是底部 Tab，而是首页和晨间页的主要入口。原因：中老年用户不应在底部看到太多抽象导航；小助手应作为“人”出现，而不是一个功能页标签。

底部导航要求：

- 高度：116dp。
- 背景：`#FFF6E2`，圆角 28dp，边框 `#F1E1BE`。
- 每个 Tab 是 1/3 宽。
- 当前 Tab：填充对应主色浅渐变，文字白色或深色高对比。
- 非当前 Tab：白色或浅灰，文字 `#1C2D49`。
- Tab 文案不超过 2 个字：`守护`、`早安`、`我的`。

## 6. 页面一：守护首页（未开始）

### 6.1 页面目标

让用户一眼知道：现在可以睡觉，点一个按钮即可开始守护。

### 6.2 第一屏布局

从上到下：

1. 顶部睡前插画卡。
2. 状态卡：`今晚安心入睡`。
3. 主按钮：`开始守护`。
4. 两个安全状态卡：`麦克风可用`、`未设家人电话`。
5. 小助手入口卡：`温柔姐姐在，您安心睡`。
6. 底部导航。

第一屏不得出现睡眠记录、检测可信度、技术说明、导出、设置列表。

### 6.3 具体视觉

背景：`#FFF9EB`。

顶部插画卡：

- 高度：300-360dp。
- 圆角：28dp。
- 背景图：`ui_sleep_scene`，填满宽度，顶部可覆盖少量深蓝渐变。
- 标题：`睡了么`，34sp，白色或深蓝，放在插画上方居中。
- 副标题：`离线守护 · 晨间关怀` 或联网版本 `睡眠守护 · 晨间关怀`，20sp。

状态卡：

- 背景：绿色浅卡。
- 文案：`守护已就绪`。
- 图标：绿色盾牌。
- 高度：72dp。

主按钮：

- 文案：`▶ 开始守护`。
- 背景：蓝色渐变。
- 高度：96dp。
- 字号：30sp。
- 左侧播放图标必须是白色圆形内蓝色三角，或直接用文字三角但要垂直居中。

安全状态卡：

- 两列。
- 高度：72dp。
- `麦克风可用` 用绿色。
- `未设家人电话` 用橙色，并带感叹号。
- 点击 `未设家人电话` 进入紧急联系人设置。

小助手入口：

- 头像左侧 96dp，文字右侧。
- 标题：`温柔姐姐在`。
- 正文：`您睡吧，我安静守着。`
- 按钮：`和我说说`，绿色软按钮。

### 6.4 功能行为

- 点击 `开始守护`：
  - 若缺麦克风权限，弹系统权限，然后回到首页。
  - 权限已允许，启动 `SleepMonitorService`。
  - 页面立即切换到“守护中”态。
- 点击 `未设家人电话`：打开紧急联系人弹窗。
- 点击小助手卡：进入实时小助手页。

### 6.5 验收标准

- 模拟器 1080x2400 截图第一屏必须看见完整 `开始守护` 按钮。
- 第一屏可点击项数量不得超过 4 个：开始守护、麦克风、家人电话、小助手。
- 点击开始守护后，`dumpsys activity services` 必须看到 `SleepMonitorService`，并且 `isForeground=true`。
- 前台通知通道必须静默：`sound=null` 或 `defaults=0`，不得播放提示音。

## 7. 页面二：守护首页（进行中）

### 7.1 页面目标

让用户知道守护正在运行，并且保持安静。

### 7.2 第一屏布局

1. 顶部插画卡，画面可以变暗一点。
2. 状态卡：`正在静静守着`。
3. 主按钮：`停止守护`。
4. 安全状态卡：麦克风、家人电话。
5. 小助手入口卡。

### 7.3 具体视觉

状态标题：`正在静静守着`，34sp，深蓝。

副标题：`没有异常时保持安静，有风险时再温和叫醒。`，22sp，灰蓝。

主按钮：

- 红色渐变。
- 高度 96dp。
- 文案：`停止守护`。
- 不要使用“暂停守护”，避免歧义。

### 7.4 功能行为

- 点击 `停止守护`：停止服务，回到未开始态。
- 进入 `早安`：自动停止守护，并 toast `已进入晨间护理`。

### 7.5 验收标准

- 守护中页面第一屏必须明显看到 `停止守护`。
- 后台服务运行时，状态栏可显示麦克风隐私指示，但不得有本 App 提示音。
- 从守护中进入早安页后，服务必须停止。

## 8. 页面三：早安护理

### 8.1 页面目标

醒来后不用读很多字，先完成喝水和吃药，再看睡眠汇报。

### 8.2 第一屏布局

1. 顶部晨间场景卡。
2. 小助手问候卡。
3. 两张大操作卡：`我喝水了`、`已吃药`。
4. 睡眠汇报卡的标题露出一部分。
5. 底部导航。

### 8.3 具体视觉

顶部晨间卡：

- 背景：`ui_morning_scene` 或晨间橙浅卡。
- 标题：`早安护理`，34sp，橙色。
- 副标题：`先喝水、确认吃药，再慢慢看昨晚报告。`

小助手问候卡：

- 左侧头像 108dp。
- 标题：`早安，温柔姐姐` 或用户给出的名字。
- 正文不超过 3 行：`早安，昨晚辛苦了。先慢慢坐起来，我们看个简单总结。`
- 按钮：`和我聊聊`，绿色。

大操作卡：

- `我喝水了`：绿色浅卡，水杯图标，按钮式整卡可点，高度 120dp。
- `已吃药`：橙色浅卡，胶囊图标，按钮式整卡可点，高度 120dp。
- 二者必须同屏可见。

睡眠汇报：

- 标题：`睡眠汇报`，蓝色。
- 第一屏只显示摘要，不显示技术细节。
- 技术细节放入 `查看详细记录`。

### 8.4 功能行为

- 点击 `我喝水了`：本地记录当天喝水确认，按钮变为 `已喝水`，绿色加深。
- 点击 `已吃药`：本地记录当天吃药确认，按钮变为 `今天已吃药`。
- 若未设置吃药提醒，第二张卡改为 `设置吃药提醒`，橙色。
- 若用户没有确认吃药，到设定时间后重复提醒。

### 8.5 验收标准

- 第一屏必须同时看见 `我喝水了` 和 `已吃药` 或 `设置吃药提醒`。
- 页面不得先展示长篇睡眠报告。
- 点击已吃药后，重新进入早安页必须保持已确认状态。

## 9. 页面四：实时小助手

### 9.1 页面目标

像 Gemini Live 一样，打开即可面对面说话。用户不用找输入框，不用点发送。

### 9.2 第一屏布局

1. 小助手半身形象大卡。
2. 小助手说话气泡。
3. 底部两枚按钮：`暂停`、`看一眼`。
4. 轻量状态文本：`我在听您说话`。

不得出现：

- 发送按钮。
- 大段说明。
- “AI”“大模型”“Token”“联网配置”。
- 超过 3 个并列按钮。

### 9.3 首次认识流程

首次打开时，小助手必须自然说：

`您好，我想先和您正式认识。以后我叫什么？我像什么身份陪您？我该怎么称呼您？`

更口语化版本：

`您好，我先认认您。您给我起个名字吧，我以后怎么称呼您？`

小助手应通过语音识别用户回答，并保存：

- 小助手名字。
- 小助手身份，例如孙女、女儿、小妹、小弟、护工、朋友。
- 对用户称呼，例如奶奶、爷爷、阿姨、叔叔、妈妈、爸爸。

如果没听清，必须回答：

`我刚才没听清，您慢慢说，不着急。`

### 9.4 具体视觉

背景：`#FFF9EB`。

小助手大卡：

- 圆角 28dp。
- 浅绿色/浅粉/浅紫渐变，随角色变化。
- 头像高度 420-520dp，占第一屏 45%-55%。
- 人物必须正面，眼睛看向用户。

说话气泡：

- 背景：角色色的 12%-18% 浅色混合。
- 圆角：24dp。
- 文本：22-26sp，Bold。
- 单屏最多 5 行，长回复要分段或语音播报，不要塞满。

按钮：

- `暂停`：橙色软按钮。
- `看一眼`：绿色软按钮。
- 如果必须保留 `我来说`，只能在语音识别失败后出现，不得默认占位。

### 9.5 功能行为

- 页面进入 350ms 内启动实时语音监听。
- 用户说话时打断 TTS。
- 小助手回答前必须先给即时安抚：
  - 找东西：`您别急，我想想。`
  - 健康不适：`您先慢慢坐好，我陪您一起看。`
  - 网络等待：`我在想，您稍等一下。`
- 回答必须短，不超过 4 句。复杂建议放到后续追问。

### 9.6 验收标准

- 打开页面第一屏必须看到小助手大头像。
- 不得出现文字输入框和发送按钮。
- 默认可见按钮不得超过 2 个。
- 模拟器无中文语音包时允许识别失败，但页面不能崩溃。
- 真机验收必须验证：用户说话可打断小助手播报。

## 10. 页面五：小助手看看

### 10.1 页面目标

让小助手通过摄像头帮助老人看气色、看药盒、读字、找东西、记位置。

### 10.2 第一屏布局

1. 标题：`小助手看看`。
2. 小助手说明卡。
3. 两个主按钮：`看我气色`、`帮我找东西`。
4. 记忆卡：`我记得的位置`。

复杂入口放到滚动下方：

- `看看药吃了吗`
- `帮我读书/说明书`
- `记住东西放哪里`
- `看看外面`
- `清除位置记忆`

### 10.3 具体视觉

小助手说明卡：

- 左头像 96dp。
- 标题：`我陪你看 · 温柔姐姐`。
- 正文：`我可以看药盒、记位置、找东西、读大字。`

主按钮：

- `看我气色`：绿色，高度 88dp。
- `帮我找东西`：蓝色，高度 88dp。
- 两个按钮必须第一屏可见。

说明文字必须短：

`只在您打开时看，不在后台偷拍。`

### 10.4 功能行为

- 点击视觉任务必须先检查摄像头权限。
- 有权限后拉起系统相机或内置相机。
- 图片最大边不超过 960px。
- JPEG 压缩后不超过 220KB。
- 自动记忆只保存结构化文字：物品、位置、时间、置信度，不保存无关隐私图。
- 典型物品：钥匙、手机、药盒、眼镜、钱包、存折、充电器。

### 10.5 验收标准

- 点击 `看我气色` 后，`dumpsys window` 必须显示系统相机或内置相机 Activity。
- 页面必须明确显示“不在后台偷拍”。
- 物品位置记忆必须能在无网络时本地检索。

## 11. 页面六：强唤醒

### 11.1 页面目标

睡眠风险发生时，最大概率叫醒用户，且用户醒来后能一键确认。

### 11.2 第一屏布局

1. 强唤醒场景图：`ui_alarm_scene`。
2. 大标题：`检测到高风险睡眠异常`。
3. 小助手安抚语。
4. 倒计时。
5. 主按钮：`我醒了`。
6. 次按钮：`我没事`、`结束本晚`。

### 11.3 具体视觉

背景：`#FFF9EB`。

标题：

- `检测到高风险睡眠异常`，32sp，红色，Bold。
- 不得写 `呼吸暂停已发生`、`诊断为`。

主按钮：

- 红色。
- 高度：104dp。
- 文案：`❤ 我醒了`。
- 字号：32sp。

倒计时：

- 26sp，深蓝。
- 演练模式必须橙色显示：`演练模式：不打电话或发短信。`

### 11.4 功能行为

- 强唤醒必须播放自定义音乐/系统闹铃并振动。
- 用户说 `我没事` 或点击按钮必须停止声音、振动、语音识别。
- 倒计时结束才触发紧急通知。
- 演练模式永远不得打电话或发短信。

### 11.5 验收标准

- 演练截图必须看到 `我醒了`、`我没事`、`结束本晚`。
- 演练模式必须显示 `不打电话或发短信`。
- 点击 `我醒了` 后 Activity 关闭，声音停止。

## 12. 页面七：我的 / 设置

### 12.1 页面目标

设置只给家属或熟练用户使用，不干扰老人日常。

### 12.2 第一屏布局

1. 标题：`我的`。
2. 安全状态卡：家人电话、唤醒声音、蓝牙音箱。
3. 小助手卡。
4. 常用设置入口：`紧急联系人`、`唤醒声音`、`蓝牙音箱`。

高级设置放下方：

- 检测模式。
- 异常证据。
- 主人档案。
- 睡前自检。
- 检测测试。
- 导出证据摘要。
- 删除本地数据。

### 12.3 具体视觉

设置页不使用密集列表，每个入口都是 80dp 高的大按钮或卡片。

安全状态卡三列或纵向：

- `家人电话`：未设置橙色，已设置绿色。
- `唤醒声音`：未设置蓝色，已设置绿色。
- `蓝牙音箱`：未连接橙色，已连接绿色。

### 12.4 功能行为

- 进入设置时不自动停止守护，除非进入早安页。
- 修改紧急联系人后必须检查电话/短信权限。
- 蓝牙音箱入口打开系统蓝牙设置。

### 12.5 验收标准

- 设置页第一屏不得出现超过 6 个入口。
- 删除数据必须二次确认。
- 紧急联系人弹窗必须说明“号码只保存在本机”。

## 13. 弹窗规范

弹窗只用于必须确认的事项：

- 紧急联系人。
- 删除本地数据。
- 权限解释。
- 找东西输入名称。

弹窗按钮：

- 确认按钮在右侧。
- 取消按钮在左侧。
- 字号不小于 20sp。

不得用弹窗做复杂 onboarding。第一次设置尽量由小助手语音自然完成。

## 14. 原生实现方法

### 14.1 代码结构

现有 Java 原生 UI 可继续使用，但必须拆出页面构建方法，避免 `MainActivity` 继续膨胀。

建议结构：

```text
MainActivity.java
  只负责导航、权限、生命周期

ui/HomeScreen.java
  buildIdleHome()
  buildRunningHome()

ui/MorningScreen.java
  buildMorningCare()

ui/CompanionScreen.java
  buildLiveChat()
  buildVision()

ui/AlarmScreen.java
  buildAlarm()

ui/SettingsScreen.java
  buildSettings()

ui/components/
  BigButton.java
  SoftCard.java
  AssistantCard.java
  StatusPill.java
  BottomNav.java
```

如果短期不拆文件，至少要在 `MainActivity` 内按页面块重排，禁止在一个方法里同时塞页面、权限、数据库、网络调用。

### 14.2 组件实现

必须实现以下组件，并在所有页面复用：

- `PrimaryActionButton(text, color, icon)`：大主按钮。
- `SoftActionCard(title, subtitle, icon, color)`：晨间和状态卡。
- `HeroScene(image, title, subtitle, palette)`：首页和晨间顶部。
- `AssistantPresenceCard(role, name, line, actionText)`：小助手入口。
- `LiveAssistantStage(role, status, speech)`：实时聊天首屏。
- `SafetyChip(label, state, action)`：麦克风、家人电话、蓝牙。

### 14.3 图片加载

所有设计模型资产必须放到：

```text
android-app/src/main/res/drawable-nodpi/
```

命名必须全小写，下划线：

```text
ui_brand_logo.png
ui_sleep_scene.png
ui_morning_scene.png
ui_alarm_scene.png
assistant_sister_live.png
```

禁止使用中文文件名作为 Android 资源名。

### 14.4 状态驱动

页面必须由明确状态驱动：

```text
HomeState.IDLE
HomeState.RUNNING
MorningState.NEEDS_MEDICATION
MorningState.MEDICATION_DONE
AssistantState.LISTENING
AssistantState.THINKING
AssistantState.SPEAKING
AlarmState.DRILL
AlarmState.ACTIVE
```

不得只靠按钮文本反推状态。

## 15. 自动验收标准

每次 UI 重做后必须跑以下验收。

### 15.1 构建验收

```powershell
cd D:\www\睡了么\android-app
powershell -ExecutionPolicy Bypass -File .\test.ps1
```

必须通过：

- APK 构建成功。
- 签名验证通过。
- 权限清单完整。
- DeepSeek 测试 Key 未打包进 APK。

### 15.2 模拟器验收

必须生成截图：

```text
build/emulator-test-home-idle.png
build/emulator-test-home-running.png
build/emulator-test-morning.png
build/emulator-test-companion-live.png
build/emulator-test-vision.png
build/emulator-test-alarm-drill.png
build/emulator-test-settings.png
```

每张截图必须人工或脚本检查以下文本：

| 截图 | 必须包含 | 不得包含 |
| --- | --- | --- |
| home-idle | 开始守护、守护已就绪、未设家人电话 | 检测可信度长文、导出、删除数据 |
| home-running | 正在静静守着、停止守护 | 开始守护 |
| morning | 早安护理、我喝水了、已吃药、睡眠汇报 | 长篇设置列表 |
| companion-live | 小助手名字、我在听、暂停、看一眼 | 发送、输入框、AI |
| vision | 小助手看看、看我气色、帮我找东西、不在后台偷拍 | 后台偷拍 |
| alarm-drill | 检测到高风险睡眠异常、我醒了、我没事、演练模式 | 诊断、确诊 |
| settings | 我的、家人电话、唤醒声音、蓝牙音箱 | 第一屏超过 6 个入口 |

### 15.3 服务验收

开始守护后必须满足：

```text
SleepMonitorService isForeground=true
foregroundNoti channel=sleep_guard_silent_v2
sound=null 或 defaults=0
```

进入早安页后必须满足：

```text
SleepMonitorService 不再运行
```

### 15.4 真机验收

模拟器通过不代表最终通过。真机必须额外验：

- 中文语音识别能启动。
- 用户说话能打断小助手播报。
- 强唤醒声音足够大。
- 蓝牙音箱连接后声音走蓝牙。
- 电话/短信权限在真实系统上可申请、可拒绝、可恢复。
- 摄像头拍照后能返回 App 并显示预览。

## 16. 返工规则

以下情况必须返工，不允许解释为“可接受”：

1. 首页第一屏出现设置列表。
2. 首页第一屏看不到完整开始守护按钮。
3. 小助手页出现发送按钮或文字输入框。
4. 晨间页第一屏看不到喝水和吃药。
5. 强唤醒页主按钮不够大或不是红色。
6. 守护服务运行时有提示音。
7. 截图风格明显不像参考图：颜色灰暗、字体小、按钮小、插画缺失。
8. 使用整张参考图截图作为页面内容。
9. 健康文案出现诊断承诺。
10. 删除数据、电话短信等高风险操作没有确认或说明。

## 17. 最终交付定义

UI/UX 重做完成不是“能打开页面”。必须同时满足：

- 每个页面通过截图验收。
- 每个核心按钮通过点击验收。
- 服务状态和通知静默通过 adb 验收。
- 参考图气质达标：温馨、可爱、明亮、大字、大按钮。
- 中老年场景达标：睡前一键开始，醒后一键喝水吃药，小助手打开就能说。

只有以上全部通过，才能进入下一轮 APK 打包交付。
