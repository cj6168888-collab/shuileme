# 睡了么当前“假功能/弱实现”审计

日期：2026-05-16

标准：用户界面或文案让人以为能力已经完整可用，但代码实际只是模拟、手动录入、系统跳转、静态估算、兜底话术或工程测试。

## 结论

当前 APK 已有可运行骨架：前台静音守护、麦克风采样、动作阈值、异常录音片段、强唤醒、语音停止、电话/短信紧急通知、服务端账号/档案/消息基础接口都不是纯空壳。

但离“可信产品体验”还差一批关键真能力。最需要先清理的是睡眠分析、设备接入、蓝牙音箱、AI 视觉/聊天、短信验证码、拟人语音和 3D 小助手这些地方的假完整感。

## P0：必须优先修正

### 1. 睡眠分析像成品，实际是规则估算

证据：
- `SleepMonitorService` 只用 RMS 声音阈值和加速度动作阈值触发事件：`mediumRmsThreshold()`、`highRmsThreshold()`、`mediumMotionThreshold()`、`highMotionThreshold()`。
- `MainActivity` 的“熟睡估算”是按总时长减扰动后乘以 `0.58`：`estimatedDeepSleepMinutes = stableMinutes * 0.58f`。
- 波形图是 `SleepWaveformView` 按事件数量绘制的示意图，不是真实整夜声学/动作曲线。

风险：
- 用户会误以为已经能判断深睡、浅睡、睡眠分期。
- 对老人健康场景，误导风险高。

处理：
- UI 文案改成“守护记录 / 波动图 / 熟睡粗估”，不要叫专业“睡眠分析”。
- 报告页增加可信度等级：手机估算、设备佐证、需复盘。
- 后续接 Health Connect/可穿戴数据后再提高可信度。

### 2. 设备接入是假接入

证据：
- “蓝牙音箱”入口只是打开系统蓝牙设置：`Settings.ACTION_BLUETOOTH_SETTINGS`。
- 心率、血氧、呼吸率是手动录入：`showDeviceReadingDialog()` / `insertDeviceReading(...)`。
- 页面明确写着“后续接入蓝牙/Health Connect”。

风险：
- 用户以为已连接血氧仪、手表、蓝牙音箱，实际没有自动读取，也没有音频输出路由确认。

处理：
- UI 改成“添加设备记录”或“打开蓝牙设置”，不能写“已接入”。
- 优先做蓝牙音箱输出检测：当前输出设备、测试音、断连回退。
- 再做 Health Connect 读取心率/血氧/呼吸率。

本轮进展：
- 已新增 `AudioOutputStatus`，读取 Android 当前音频输出路由，识别蓝牙 A2DP/SCO、BLE、助听器、有线耳机和手机扬声器。
- 设置页已显示“蓝牙已连/蓝牙未连”，唤醒输出页显示当前路由、闹钟音量、测试音、系统蓝牙设置入口和断连回退说明。
- 睡前自检和强唤醒页已显示实际输出路径；模拟器验证为“未检测到蓝牙音箱，回退手机扬声器和震动”。
- 仍未完成：真实蓝牙音箱真机连接/断连场景、Health Connect、血氧仪/手表原始数据读取。

### 3. AI 能力存在兜底假回答

证据：
- 服务端已改为阿里百炼/DashScope 多模态优先：`/api/chat` 走 `ALIYUN_TEXT_MODEL`，`/api/vision` 走 `ALIYUN_VISION_MODEL`，`/api/audio` 走 `ALIYUN_AUDIO_MODEL` 或声波摘要分析。
- APK 睡眠报告已接入 `/api/audio`：会上传真实采样统计、异常摘要和设备摘要；如果有正式异常 WAV 片段且大小可控，会附带一段现场录音，让小助手生成睡眠声波复盘。未登录或未配置模型时明确提示，不伪造模型分析。
- APK 设置页已接入服务能力自检：从服务端 `/health` 读取短信、阿里多模态、数字人配置状态；未配置阿里 Key 时直接显示兜底状态，不再隐藏成“已联网 AI”。
- 管理后台已接入 `/api/admin/model/probe`：需要管理令牌，手动点击才会调用一条短文本探针验证阿里模型真实可用；未配置 Key 时返回兜底状态，不伪造在线模型。
- 服务端新增 `npm run smoke:model`：用临时数据库真实调用 `/api/chat`、`/api/vision`、`/api/audio`，确认文本、看图和声波摘要都实际走阿里模型；音频冒烟会附带一段真实 WAV base64，验证异常录音片段上传链路；未配置 Key 时跳过。
- 服务端消息已从“只发通知”升级为“小助手开口读”：APK 后台轮询到待朗读消息时，非睡眠守护状态会拉起主动关怀页直接朗读并可确认已读；睡眠守护中普通消息不打扰，紧急消息才走全屏语音兜底。
- 服务端启动会读取桌面 `guanlin-aliyun-ai.env`，其中的 `DASHSCOPE_API_KEY` 可直接作为阿里百炼/DashScope 模型 Key；未配置 `DASHSCOPE_API_KEY` / `ALIYUN_MODEL_API_KEY` / `ALIYUN_BAILIAN_API_KEY` 时，聊天、看图、听音都会明确返回兜底状态，不假装已经真实联网、多模态识别或看图。
- DeepSeek 仅保留为文字兜底，不再作为视觉、音频或数字人方案。
- APK 侧未登录服务端时，多处返回“模型服务还没配置好”的本机兜底。

风险：
- UI 如果显示“AI 已分析/我看完了”，但实际是兜底话术，会严重伤害信任。

处理：
- 服务端返回字段已加 `model_used/model_provider/model_name`、`vision_used/vision_provider/vision_model_name`、`audio_used/audio_provider/audio_model_name`。
- APK 根据字段显示“AI分析完成”或“服务未连上，本机建议”。
- 视觉失败时不能说“看完了”。

### 4. 手机号验证码是开发模式

证据：
- `sendSmsCode()` 默认 `GOUXIONG_DEV_SMS=1`，直接返回 `dev_code`。
- APK 验证码输入框会显示“测试验证码”。

风险：
- 不能作为真实用户系统上线。

处理：
- 接真实短信服务商。
- 生产环境禁止返回 `dev_code`。
- 管理后台加用户删除、导出、审计。

本轮进展：
- 已接入阿里云短信 `SendSms` 服务端发送，配置来自 `server/.env`、项目根 `.env` 或桌面 `aliyun-sms.env`，Key 不进入 APK。
- 服务端健康检查返回短信状态：`dev`、`aliyun` 或 `missing`，不暴露密钥。
- `GOUXIONG_DEV_SMS=1` 仅用于本地测试；真实阿里云配置完整且未启用开发模式时，验证码接口不会返回 `dev_code`。
- 验证码接口已加同手机号、同来源限流，超限返回 `429` 和重试秒数，避免真实短信被刷。
- APK 验证码弹窗已区分开发短信和真实短信，真实短信模式下提示用户查看短信，不再说“本地开发模式会显示测试验证码”。
- 已补服务端用户自助导出/删除、管理后台导出/删除；导出不包含验证码哈希、会话哈希或服务端密钥。
- 已补管理后台审计日志：查看详情、导出、删除、消息下发、模型探测、用户自助导出/删除都会记录，且不记录验证码或密钥。
- 已补聊天长期记忆自动提取：聊天原文进入 `insights`，结构化主人画像进入 `memory_items`，覆盖身体、用药、睡眠、家庭、兴趣、情绪、财务风险、异常事件和偏好，并进入后续模型上下文。
- 已补主动关怀专用接口：`/api/care/brief` 根据喝水、吃药、晨间简报、睡眠异常唤醒等场景生成可直接朗读的话术，可选写入 `push_messages`，APK 主动关怀页已改走该接口。
- 仍未完成：线上短信模板签名审核状态、图形验证码/风控、备份和误删恢复流程。

## P1：影响体验真实性

### 5. “小智音色”仍不是完整小智语音资产

证据：
- `XiaozhiVoiceProfile` 仍只是 Android `TextToSpeech` 的音色、语速、音高选择，不能保证每台手机都有符合“懂事小弟/阳光小哥”的男声。
- 已新增 `/api/live/session` WebSocket、PCM16 音频帧传输、阿里 Realtime 桥接、模型音频帧回传、APK `AudioTrack` 低延迟播放和 `abort` 中断转发，但这仍不是直接复用开源“小智”的全部语音资产。
- 当前真实链路以阿里 Realtime + 系统 TTS/AudioTrack 为主，小智资产只作为体验参考和文档路线，不应在 UI 里写成“小智音色已完整接入”。

风险：
- 男孩/男青年仍可能是女声，取决于系统 TTS voice。
- 如果把当前音色包装成“小智拟人语音”，用户会误以为已经具备稳定真人音色和统一跨机型表现。

处理：
- UI 文案不要写“小智音色已接入”。
- 已接服务端流式音频链路时，文案应写“阿里 Realtime/系统语音输出已启用”，未接通时明确“系统语音包”。
- 对四个角色做真实 voice id 配置。

### 6. 3D 小助手是动画舞台，不是真 3D

证据：
- 聊天页默认使用原生 `ImageView` + 缩放/摆动/点阵动画，已避免 Live 首屏默认启动重 WebView 导致模拟器 ANR。
- 没有 Live2D、VRM、骨骼、口型同步。

风险：
- 用户要求“像活的真人陪伴”，当前只能算可爱的动效壳。

处理：
- 短期：明确为“动态小助手”。
- 中期：接 Live2D/TalkingHead/VRM 任一路线，并做口型、眨眼、表情状态。

### 7. 实时聊天已接近 Live 链路，但还不是 Gemini Live 级体验

证据：
- APK 仍保留 Android `SpeechRecognizer` 分段识别作为兜底；已增加基于实时 PCM RMS 的保守自动插话检测，并从单一固定阈值升级为 30ms 短帧、自适应噪声底、回落阈值和 240ms 最短语音门，但真实麦克风插话阈值、回音消除和长时稳定需要真机调优。
- 已实现的 Live 链路是：`LivePcmRecorder` 发送 16kHz/30ms raw PCM16 -> `/api/live/session` -> 可选阿里 Realtime `input_audio_buffer.append` -> 服务端回传 `stt`、`sentence_delta` 和二进制模型音频帧 -> APK `LivePcmPlayer` 使用 `AudioTrack` 播放。
- 用户点击小助手舞台或触发中断时，APK 会停止本地 TTS/PCM 播放并发送 `abort`；服务端已向 Realtime 桥转发 `response.cancel` 和 `input_audio_buffer.clear`。
- 小助手说话或播放模型音频期间，APK 会在连续多帧 RMS 超过动态阈值时记录 `last_live_auto_barge_in_count`、`last_live_auto_barge_in_speech_ms`、动态阈值和噪声底，并触发同一条 `abort` 链路；当前自动化能验证处理链路，不能替代真实扬声器回采环境。
- 自动化已验证本地假 Realtime 端收到音频帧、取消和清空缓冲；`smoke:realtime` 已能用真实阿里 Key 建立 `session.created`。

风险：
- 模拟器验收不能替代老人真机环境：扬声器回采、方言、弱网、系统后台限制都会影响“像真人面对面”的感觉。
- UI 如果写“Gemini Live 级”或“真人数字人”，仍属于夸大。

处理：
- 设置页按 `/health` 显示：Realtime 桥、服务端 ASR、模型音频回传、APK 低延迟播放分别是否启用。
- 文案可写“Live 实时陪伴链路已启用”，但要附带“仍需真机验证回音消除和长时稳定”。
- 下一步优先做真机回音消除和自动插话阈值标定、Opus 或确认 PCM16 长链路成本和稳定性。

### 8. 主动关怀还是通知/Activity，不是真正后台自然喊话

证据：
- 喝水、吃药由 `AlarmManager` 触发 `Notification` 和 `ProactiveCareActivity`。
- Android 后台直接拉起 Activity 会受系统限制。

风险：
- 不是所有机型都能像真人一样直接叫用户。

处理：
- 做“可用性分级”：前台可直接喊，后台用高优先级通知/全屏场景，失败时退化。
- 引导用户开启自启动、锁屏显示、后台弹出。

## P2：应降级或隐藏的演示功能

### 9. 检测测试入口太像产品功能

证据：
- `showDetectionTest()` 下有“模拟鼾声”“模拟咳嗽/轻喘”“模拟尖叫/强动作”“填入模拟手表摘要”。
- `simulateEvent()` 会生成 demo WAV 并写入记录。

风险：
- 普通用户可能把模拟记录当真实记录。

处理：
- 移入“开发者测试”或长按入口。
- 所有模拟事件加明显 `演练` 水印，不进入默认睡眠报告统计。

### 10. 早安/起床判断很粗

证据：
- `maybePromptMorningCare()` 只按守护时长、清晨时间窗、动作评分判断“可能醒来”。

风险：
- 误报早安，影响睡眠。

处理：
- UI 叫“可能醒了”，不要说“系统检测到起床”。
- 后续结合充电状态、屏幕点亮、位置移动、音频模式、手表睡眠结束。

## 已经是真能力但需要继续加强

- 静音前台守护：已经设置低重要性通知通道，不会发运行提示音。
- 异常录音：能保存触发前后短 WAV，但不是整夜录音。
- 强唤醒：全屏 Activity + 闹钟/自定义声音 + 震动 + 语音停止。
- 紧急联系人：能自动短信/电话；无权限时回退系统短信/拨号页。
- 服务端基础：有手机号会话、档案、洞察、聊天代理、视觉代理、消息轮询和管理后台。

## 下一步建议

1. 先改 UI 文案和入口，把假完整感降下来。
2. 隐藏模拟测试，避免污染正式报告。
3. 给 AI/视觉/设备/短信加真实状态标记。
4. 优先实现蓝牙音箱输出检测和 Health Connect 读取。
5. 再升级实时语音链路和小助手动效。
