# 2026-05-23 全局真实性检查

原则：没有真实输入、真实输出、失败态、用户可见状态和验收记录，就不能显示“完成/可用/通过”。

## 本轮发现并修正

| 能力 | 原风险 | 本轮处理 | 当前真实状态 |
| --- | --- | --- | --- |
| 模型能力 | Key 配好容易被 UI 说成“模型可用” | 诚实检查改为“Key已配，需探测” | 配置存在不等于调用成功 |
| 实时语音 | 服务端 Realtime 配置容易被说成“模型音频优先” | 诚实检查改为“链路已配，真机待验” | 代码链路存在，真机长时语音待验 |
| 语音听懂 | 拾音波形曾被误当成听懂人话 | 已拆成“麦克风”和“听懂人话” | SpeechRecognizer 返回文字才算听懂 |
| 睡眠守护 | 服务启动曾被误当成真实识别 | 已记录守护 AudioRecord 启动、读帧、RMS、峰值、错误 | 模拟器证明读帧，真机触发率待验 |
| 助眠音乐 | “放音乐”容易误解为版权音乐平台 | 文档明确本地雨声/白噪声，不是歌曲库 | 本地合成音已接入，音乐平台未接入 |
| 新闻 | 模型可能编新闻 | 无 RSS 时保持未接入，不编新闻 | 配置 RSS 后才读来源条目 |
| Avatar | 本机 2D 可能被说成 Live2D | 健康状态继续标 `live2d_sdk=false` | 原生 2D 状态机，不是 Live2D |
| 医疗能力 | 睡眠/声波容易被理解成诊断 | 文档和提示继续限定生活提醒 | 不做诊断，不判断疾病 |

## 真机验收失败后的追加检查

| 问题 | 风险 | 处理 |
| --- | --- | --- |
| 守护拾音状态可能沿用旧记录 | 旧的读帧记录会让诚实检查误判通过 | `sleepGuardAudioPassed()` 现在要求 90 秒内有新读帧，否则显示“过期，需重测” |
| 前台麦克风服务可能被真机系统拦截 | UI 显示已开始，但服务实际失败 | `startForeground` 和 `startForegroundService` 失败会写入“服务启动失败/前台守护启动失败” |
| “您睡了么？”确认只存在内存 | 真机锁屏/重建进程后，用户回答“没睡”可能被当普通聊天 | 睡眠确认状态会从最近 `last_voice_state_stage=sleep_check` 恢复，5 分钟内仍能识别“没睡/继续” |

## 当前必须保留的“未证明”

- 真实老人弱声、方言、电视背景音下的语音识别准确率。
- 真机后台长时间守护、厂商电池策略、锁屏全屏唤醒可靠性。
- 睡眠守护对鼾声、咳嗽、喘息、说话、环境噪声的触发率和误报率。
- 阿里 Realtime 模型音频在真机上的长时稳定、回音消除、插话打断效果。
- Live2D SDK、版权音乐平台、权威实时新闻源。
- 医学级睡眠分期、睡眠呼吸暂停诊断、用药决策。

## 已补的运行时闸门

- 服务端 `/health` 新增 `truth` 字段，列出每个高风险能力的证明状态和未证明边界。
- APK 诚实检查继续显示“麦克风”“听懂人话”“睡眠守护拾音”等分层状态。
- 睡眠守护服务把真实读帧状态写入本地偏好，避免只凭服务启动宣称通过。

## 验收结论

- 自动化只能证明代码路径和模拟器读帧，不等于真机完成。
- 下一步必须用真机样本验收睡眠守护声音触发，失败就继续降级显示，不允许再用“守护已完成”类文案。
