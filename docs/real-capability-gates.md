# 真实能力闸门

目标：睡了么可以快速加功能，但不能让界面、话术或文档说得比真实代码更强。

## 基本规则

每个新能力进入主界面前，必须写清楚：

- 能力名
- 真实输入
- 真实输出
- 兜底行为
- 用户可见状态
- 不能宣传成什么
- 自动化验收
- 真机验收

没有真实输入源，就不能让模型自由编。没有播放器，就不能显示正在播放。没有实时新闻源，就不能说“今天新闻”。

## 当前能力状态

### 全局真实性闸门

- 真实输入：每项能力必须能指出来自用户、设备、服务端或文件的真实输入。
- 真实输出：必须有代码路径产生用户可感知输出，或明确显示未接入。
- 兜底行为：兜底只能说“本地/未接入/待验”，不能伪装成云端、新闻、音乐平台或医学判断。
- 用户可见状态：APK 诚实检查和服务端 `/health.truth` 必须暴露证明状态。
- 不能宣传成：只配置了 Key 就说模型已通过；只启动服务就说睡眠识别已完成；只采到波形就说听懂人话；本地雨声说成歌曲/音乐平台；本机 2D 说成 Live2D。
- 自动化验收：`android-app/test.ps1`、`android-app/e2e-live-voice.ps1`、`server/test-server.mjs` 必须通过；新增能力必须补失败态测试。
- 真机验收：涉及麦克风、语音识别、摄像头、全屏唤醒、蓝牙/音频路由、后台保活的能力，自动化只能算“代码路径通过”，不能算“真机通过”。

### 睡眠守护拾音

- 真实输入：`SleepMonitorService` 前台服务里的 `AudioRecord`。
- 真实输出：偏好记录 `last_sleep_guard_audio_started/reads/rms/peak/error`，睡眠信号样本表 `signal_samples`，中/高风险事件。
- 兜底行为：麦克风未授权、AudioRecord 初始化失败、读帧错误时必须写入错误状态，不允许显示守护拾音通过。
- 用户可见状态：诚实检查显示“睡眠守护拾音：读 N RMS X 峰 Y”或错误。
- 不能宣传成：医学睡眠分期、呼吸暂停诊断、跨机型稳定识别已完成。
- 自动化验收：模拟器可验证服务启动和读帧状态；不能替代真实声音触发率。
- 真机验收：必须用鼾声、咳嗽、说话、电视背景音、静音环境测试触发/误触发。

### 自然语音入口

- 真实输入：用户用语音或文字说“讲故事/听新闻/放点音乐/雨声/停音乐”等。
- 真实输出：APK 先用本地意图分类，把请求路由到睡前故事、新闻源检查、本地助眠音开始/停止等真实能力。
- 兜底行为：识别不到明确能力时才进入普通陪伴聊天；新闻和音乐仍遵守各自真实能力边界。
- 用户可见状态：服务端 `/health.companion.voice_shortcuts.implemented=true`；客户端能力页显示自然语音入口已覆盖。
- 不能宣传成：无限意图识别、智能音箱全品类技能、任意音乐/新闻聚合。
- 自动化验收：`android-app/test.ps1` 静态检查 `looksLike*Request` 与 E2E 调试触发词；`server/test-server.mjs` 检查 `/health.companion.voice_shortcuts`。
- 真机验收：老人方言、弱声、停顿、电视背景音下仍需测试触发准确率。

### 睡前故事

- 真实输入：用户点击“故事”或语音要求讲故事。
- 真实输出：已配置模型时走小助手聊天；未配置模型时使用本地固定睡前故事。
- 兜底行为：明确是本地故事，不假装联网生成。
- 用户可见状态：可以讲故事。
- 不能宣传成：海量故事库、版权有声书、个性化长篇连载。
- 自动化验收：`android-app/test.ps1` 静态检查 `askBedtimeStory` 和 `localBedtimeStory`。
- 真机验收：TTS 能完整朗读，用户插话可打断。
- 入口覆盖：按钮、文字输入和语音说“讲故事/睡前故事/哄我睡”等都必须进入同一真实能力路径。

### 本地助眠音

- 真实输入：用户点击“助眠音”或提出听轻雨声/白噪声。
- 真实输出：APK 用 `AudioTrack` 本地合成轻雨声/白噪声，可开始、停止和淡出。
- 兜底行为：不联网、不下载；版权音乐平台仍明确未接入。小助手询问“您睡了么？”时会先压低助眠音，用户说没睡后再慢慢恢复；转入守护时渐弱停止。
- 用户可见状态：本地助眠音已接入，版权音乐未接入。
- 不能宣传成：版权音乐、音乐平台、歌曲库、正在播放某首歌。
- 自动化验收：`/health.companion.music_playback.implemented=true` 且 source 为 `apk_local_audiotrack_generated_rain`；`android-app/test.ps1` 静态检查 `SleepSoundPlayer`。
- 真机验收：能播放、停止；进入睡眠守护时能淡出，不突然变大声。
- 入口覆盖：按钮、文字输入和语音说“放点音乐/雨声/白噪声/停音乐”等都必须进入本地助眠音，不假装播放版权音乐。

### 新闻

- 真实输入：用户点击“新闻”或提出听新闻。
- 真实输出：服务端配置 `GOUXIONG_NEWS_RSS_URL` 后读取 RSS 条目；未配置时只返回能力状态说明。
- 兜底行为：提示新闻源尚未接入，不让模型编新闻。
- 用户可见状态：已接入真实新闻源 / 新闻源未接入。
- 不能宣传成：未验证来源的今日新闻、实时新闻、权威新闻。
- 自动化验收：无新闻源时 `/health.companion.news_briefing.implemented=false` 且 `/api/news/brief` 返回未配置；配置真实 RSS 后才允许为 true。
- 真机验收：无新闻源时不能生成疑似新闻内容；有新闻源时必须读出来源和条目，不把模型改写当事实源。
- 入口覆盖：按钮、文字输入和语音说“听新闻/说新闻/新闻简报”等都必须先检查真实新闻源。

### 可能入睡确认

- 真实输入：语音陪伴页长时间无用户声音或系统语音识别超时。
- 真实输出：小助手轻声问“xx，您睡了么？”。
- 兜底行为：无人回应时转入静音睡眠守护；听到“没睡/继续/再讲”等语义就继续陪伴。
- 用户可见状态：可能睡着确认。
- 不能宣传成：医学入睡判断、准确睡眠分期。
- 自动化验收：`android-app/test.ps1` 静态检查 `maybeAskIfUserAsleep`、`looksLikeStillAwakeReply`、`looksLikeReadyToSleepReply`、`enterSleepGuardFromCompanion`；`e2e-live-voice.ps1` 通过 `DEBUG_SLEEP_CHECK` 和 `DEBUG_SLEEP_CHECK_TIMEOUT` 触发确认，分别验证用户回应没睡后继续陪伴、不进入守护，回应睡了后转入守护，以及无回应超时后转入守护。
- 真机验收：方言、弱声、电视背景音、长时间 TTS 后的识别效果仍需调参。

## 发布前闸门

1. `/health` 必须暴露真实能力状态。
2. UI 文案必须区分“已接入 / 本机兜底 / 未接入”。
3. 每个失败态必须有测试：无 Key、无新闻源、无播放器、无摄像头、无麦克风。
4. 演练入口不能污染正式睡眠报告。
5. 医疗、新闻、音乐、金融风险相关能力必须宁可少说，也不能编。
6. 只要真机未验收，文案必须写“真机待验”或“未完整证明”。
7. 历史文档里“已实现/可用/完成”必须按本文件复核；不能作为当前真实状态依据。
