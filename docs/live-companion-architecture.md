# 小助手 Live 实时陪伴架构

版本：V1.0  
目标：把小助手从“按屏幕聊天”升级为 Gemini Live 风格的面对面实时沟通体验。

## 1. 目标定义

用户要的不是传统聊天页，而是打开后像真人在面前陪着：

- 不需要按发送。
- 不需要看转写文字。
- 不需要每一句都点按钮。
- 用户可以随时插话打断。
- 小助手能听声音、看摄像头、记住周围物品。
- 小助手回应要快，有表情和动作，不像客服机器人。

因此产品目标改为 **Live-first Companion**：

- 语音是主入口，文字只是辅助。
- 摄像头是自然观察能力，不是单独拍照工具。
- UI 是“面对面陪伴舞台”，不是消息列表。
- AI 会先安抚，再检索、分析、回答。

## 2. 现有实现不足

当前 Android 实现更接近：

```text
SpeechRecognizer 识别一句话 -> 发送给 DeepSeek 文本模型 -> TTS 朗读
```

这个方案能验证功能，但体验上不是 Gemini Live：

- 有明显轮次感，像“说完一句再等回复”。
- 打断只是停止 TTS，不是真正全双工。
- 视频不是连续上下文，只是低频抓拍。
- 延迟高时，用户会以为软件没反应。
- 页面仍容易变成按钮和状态卡，而不是面对面交流。

结论：它只能作为兜底，不作为最终目标。

## 3. Gemini Live 风格能力拆解

需要实现的能力不是某个页面，而是一套实时会话管线：

| 能力 | 体验要求 | 工程含义 |
| --- | --- | --- |
| 持续听 | 打开就能说 | 麦克风流式采集 |
| 低延迟回应 | 像真人接话 | 模型流式音频输出 |
| 可打断 | 用户插话助手停下 | barge-in / interruption |
| 不点发送 | 自然话轮 | 服务端或模型 VAD |
| 能看见 | 摄像头理解环境 | 低清视频帧或图片流 |
| 有动作 | 头像像活的 | listening/speaking/thinking 状态驱动动画 |
| 先安抚 | 等待时不沉默 | 本地即时短句 + 模型后台处理 |
| 会记事 | 找钥匙、药盒 | 本机结构化物品记忆 |

## 4. 推荐技术路线

### 4.1 首选：Gemini Live API / 原生实时音视频

适用场景：

- 真正要接近 Gemini Live。
- 需要实时音频输入输出。
- 需要摄像头画面参与理解。
- 需要更自然的打断。

Android 路线：

- 使用支持 Live API 的 Gemini 模型。
- Android 端通过 Firebase AI Logic 或 Gemini Live API 建立实时会话。
- 麦克风 PCM 流式输入。
- 摄像头低清帧按节流策略发送。
- 模型输出音频流直接播放。
- 根据模型事件驱动头像：听、想、说、看、安抚。

注意：

- 不能把私密服务端 Key 硬编码进 APK。
- 若使用 Firebase 移动端直连，必须配置应用限制、包名限制和滥用防护。
- 夜间强唤醒和紧急联系人仍然本地兜底，不能依赖网络模型。

### 4.2 备选：OpenAI Realtime / 其他实时音频模型

如果后续决定不走 Gemini，也要用同等能力的实时模型：

- 双向音频流。
- 模型端或会话端 VAD。
- 支持打断。
- 支持图像或视频输入。
- 支持函数调用：找物品、读取睡眠摘要、设置提醒。

### 4.3 兜底：DeepSeek 文本 + 本地 ASR/TTS

仅用于：

- 没有实时模型 Key。
- 网络质量不稳定。
- 用户关闭摄像头/实时语音。
- 功能测试或低成本版本。

兜底体验必须伪装成 Live：

- 不显示“发送”。
- 不显示大段转写。
- 先播放本地安抚语：“您别急，我在听，我想想。”
- 后台再走文本模型。

## 5. Live 页面设计

页面不再以消息列表为中心。

第一屏结构：

```text
顶部：小助手名字 + 连接状态
中部：大号 3D 半身助手，面向用户
气泡：一句正在说的话或正在听的状态
底部：三个大按钮
  - 我来说 / 正在听
  - 暂停
  - 看一眼
```

禁止：

- 不放“发送”主按钮。
- 不把聊天转写铺满屏幕。
- 不把摄像头当成单独拍照流程。
- 不用红色“打断”按钮，红色只给强唤醒。

状态文案：

- 听：`我在听您说`
- 说：`我慢慢说给您听`
- 想：`您别急，我想想`
- 看：`我看一眼周围，帮您记住`
- 离线兜底：`网络慢一点，我先陪着您`

## 6. 头像状态机

小助手要像活的，需要用会话事件驱动动画。

```text
IDLE -> LISTENING -> THINKING -> SPEAKING
  |         |            |          |
  |         |            |          -> INTERRUPTED -> LISTENING
  |         |            |
  |         -> SEEING -> THINKING
  |
  -> REASSURE -> LISTENING
```

状态含义：

- `IDLE`：安静待命，轻微呼吸。
- `LISTENING`：眼神看向用户，轻点头。
- `THINKING`：安抚语 + 小幅思考动作。
- `SPEAKING`：口型或身体轻动，音频同步。
- `INTERRUPTED`：立即停说，回到倾听。
- `SEEING`：摄像头采样，显示看一眼提示。
- `REASSURE`：用户急躁或找东西时先安抚。

## 7. 摄像头策略

目标是“看见和记住”，不是让老人操作拍照。

启动规则：

- 进入 Live 聊天后，如果用户已授权摄像头，可低清自动看一眼。
- 未授权时，不弹窗打断语音；等用户说“帮我看看”或点“看一眼”再解释并申请。

采样规则：

- 分辨率优先低清，例如 640x480 或更低。
- 压缩到小图，减少延迟。
- 夜间守护中不后台拍摄。
- 只保存结构化文字记忆，不默认保存原图。

默认识别：

- 钥匙
- 手机
- 药盒/药瓶
- 眼镜
- 钱包
- 遥控器
- 水杯
- 证件/医保卡
- 拐杖

回答示例：

```text
奶奶您别急，我想起来了。我昨天看见钥匙好像挂在冰箱旁边，您先去那里看看，我也继续帮您找。
```

## 8. 首次认识

首次认识不能像注册表单。

流程：

1. 用户第一次进 Live 页。
2. 小助手直接开口：
   `您好呀，我先陪着您。您想给我起个什么名字？我该怎么称呼您？`
3. 用户自然说：
   `你叫暖暖，像女儿一样陪我，叫我奶奶。`
4. 系统提取：
   - 助手名：暖暖
   - 身份：像女儿一样陪伴
   - 用户称呼：奶奶
   - 偏好：听话、哄开心、耐心
5. 小助手确认：
   `好呀奶奶，那我以后就叫暖暖。我会听您的，慢慢陪您说话。`

## 9. 安全边界

Live 体验不能破坏夜间安全：

- 夜间守护期间不闲聊、不主动播报运行状态。
- 强唤醒不交给云端模型决定。
- 紧急联系人倒计时本地执行。
- 医疗问题不诊断，只建议记录、观察和咨询医生。
- 用药问题不改药量、不建议停药。

## 10. 实施阶段

### 阶段 A：UI 先改成 Live 舞台

- 去掉聊天页消息列表优先布局。
- 放大 3D 小助手。
- 底部只保留“我来说 / 暂停 / 看一眼”。
- 普通打断按钮改蓝绿，不用红色。
- 等待时即时播放安抚语。

### 阶段 B：本地伪 Live 兜底

- SpeechRecognizer 连续听。
- TTS 可被用户语音或按钮打断。
- DeepSeek 返回前先给本地安抚语。
- 摄像头采样和物品记忆继续保留。

### 阶段 C：接入真实实时模型

- 接 Gemini Live API 或同等实时模型。
- 麦克风流式上传。
- 音频流式播放。
- 实现真正打断。
- 摄像头帧进入实时上下文。

### 阶段 D：角色动效升级

- 每个角色 6 个姿态。
- 听、说、想、看、安抚状态切换。
- 后续再做口型和更细动效。

## 11. 验收标准

体验验收：

- 打开小助手 1 秒内进入“我在听”。
- 用户不点发送也能完成连续三轮对话。
- 小助手说话时用户插话，助手能停下。
- 后台检索超过 800ms 时，先有安抚语。
- 找东西时，先查本机记忆，再看摄像头。
- 页面第一眼像面对面陪伴，不像表单或客服聊天。

工程验收：

- 实时能力和兜底能力分层，不能混成一团。
- 夜间本地唤醒逻辑不依赖云端。
- API Key 不硬编码进 APK。
- 摄像头采样有节流和大小限制。
- 所有 Live 状态都有对应 UI 动画状态。
